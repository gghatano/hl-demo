# Hyperledger Fabric デモ環境構築 Spec

## 1. 背景

製品の受発注・譲渡履歴を、複数企業間で改ざん困難な形で記録し、後続の購入者が来歴を確認できることを示すデモ環境を構築する。

対象ユースケースは、製品 `X` が `A → B → C` と販売された際に、C が必要なタイミングで「この製品が正しく A を起点として流通してきたものであること」を確認できるようにすることである。

本デモでは、Hyperledger Fabric を用いて、企業間で共有されるトレーサビリティ台帳を構築する。

---

## 2. 目的

本デモの目的は以下の通り。

1. Hyperledger Fabric を用いて、ローカル Linux 環境で動作する最小構成のサプライチェーン・トレーサビリティデモを構築する
2. 製品の生成・移転・履歴照会ができることを示す
3. C が製品の来歴を確認し、A 由来であることを検証できることを示す
4. 正常系だけでなく、不正な移転や不整合が拒否されることを示す
5. デモ実行手順・説明資料に転用しやすい形で成果物を整理する

---

## 3. スコープ

### 3.1 対象
- Hyperledger Fabric のローカルデモ環境構築
- 最小限のネットワーク構成
- Chaincode の実装
- デモ用スクリプトまたは手順書の整備
- 正常系・異常系のシナリオ実行

### 3.2 非対象
- 本番運用向けの可用性設計
- 大規模性能試験
- 外部 ERP や在庫管理システムとの本格連携
- 実物製品とデジタル記録の物理的な真正性保証
- 高度な秘密計算やプライバシー強化技術の統合

---

## 4. デモの到達イメージ

以下の流れを実演できること。

1. A が製品 `X001` を生成する
2. A が `X001` を B に譲渡する
3. B が `X001` を C に譲渡する
4. C が `X001` の履歴を照会する
5. C が `A → B → C` の流れを確認できる
6. 不正な譲渡要求や所有者不一致の更新が拒否されることを確認する

---

### 5.1 直接利用者

- 検証担当者
- 提案・説明担当者

### 5.2 間接的な閲覧者
- 顧客向け説明を受ける業務担当者
- 技術選定を行う担当者
- サプライチェーン管理・トレーサビリティに関心のある関係者

## 6. システム概要

### 6.1 全体像
Hyperledger Fabric 上に、製品単位の来歴管理台帳を構築する。  
各組織は独立した参加者としてネットワークに参加し、製品の譲渡を台帳に記録する。

### 6.2 組織
- OrgA: メーカー A
- OrgB: 卸 B
- OrgC: 販売者 C

### 6.3 ネットワーク構成
- 1 Orderer
- 各組織 1 Peer
- 1 Channel（例: `supplychannel`）

### 6.4 データモデル
管理対象は製品単位とする。

例:
- `productId`: 製品ID
- `currentOwner`: 現所有者
- `status`: 状態
- `history`: 譲渡履歴
- `createdAt`: 登録日時
- `updatedAt`: 更新日時

---

## 7. 機能要件

### 7.1 製品登録
A が新規製品を登録できること。

#### 入力
- productId
- manufacturer
- initialOwner

#### 要件
- 既存 productId と重複した登録は不可
- 初期所有者は登録主体と一致すること
- 履歴の初回イベントを保持すること

---

### 7.2 製品移転
現在所有者が、製品を次の所有者へ移転できること。

#### 入力
- productId
- fromOwner
- toOwner

#### 要件
- `fromOwner` は現在所有者と一致すること
- 一致しない場合は拒否すること
- 移転成功時に `currentOwner` を更新すること
- 履歴に移転イベントを追記すること

---

### 7.3 製品照会
指定した製品IDに紐づく現在情報を参照できること。

#### 取得項目
- productId
- currentOwner
- manufacturer
- status
- updatedAt

---

### 7.4 履歴照会
指定した製品IDに対し、登録から現在までの譲渡履歴を取得できること。

#### 取得項目
- eventType
- fromOwner
- toOwner
- txId
- timestamp
- actor

#### 要件
- C が履歴を見て、A 起点であることを確認できること
- 表示順は時系列順であること

---

### 7.5 来歴検証補助
厳密な暗号学的検証機能まで必須とはしないが、少なくとも以下を満たすこと。

#### 要件
- 製品の初回登録主体が A であることを確認できる
- 履歴が `A → B → C` と連続していることを確認できる
- 所有者の不整合更新が拒否されることを示せる

---

## 8. 非機能要件

### 8.1 実行環境
- Linux ローカル環境
- Docker / Docker Compose 相当での起動を前提とする
- Claude Code により構築・修正可能な構成とする

### 8.2 再現性
- 初回セットアップ手順が README に整理されていること
- コマンド実行により再現できること
- 可能な限り手作業を排除すること

### 8.3 可読性
- ディレクトリ構成が明確であること
- スクリプト名・ファイル名が役割を表すこと
- デモ担当者が理解しやすい説明を付すこと

### 8.4 デモ容易性
- 5〜10分程度で主要シナリオを実演できること
- 画面または CLI 出力が、第三者に説明しやすいこと

---

## 9. デモシナリオ要件

### 9.1 正常系シナリオ
#### シナリオ N1: 製品登録
- A が `X001` を登録する
- 登録結果として currentOwner が A になる

#### シナリオ N2: A → B への移転
- A が `X001` を B に移転する
- currentOwner が B に更新される

#### シナリオ N3: B → C への移転
- B が `X001` を C に移転する
- currentOwner が C に更新される

#### シナリオ N4: C による履歴照会
- C が `X001` を照会する
- 履歴として `A → B → C` が確認できる

---

### 9.2 異常系シナリオ
#### シナリオ E1: 所有者不一致の移転要求
- A が、すでに B 所有となった `X001` を再度第三者へ移転しようとする
- 拒否されること

#### シナリオ E2: 未登録製品の照会
- 存在しない productId を照会する
- 適切なエラーとなること

#### シナリオ E3: 重複登録
- 既存の `X001` を再登録しようとする
- 拒否されること

---

## 16. 技術上の留意点

### 16.1 重要な前提

このデモで保証するのは、あくまで「台帳上の来歴の一貫性」と「参加者間で共有された記録の改ざん困難性」である。

### 16.2 限界

このデモだけでは、現実世界の物理的な製品が本当に当該 productId に対応していることまでは保証しない。

つまり、以下は別論点である。

* QRコードの貼付
* RFID タグ
* IoT センサー連携
* 真正品判定デバイス連携

必要であれば、将来拡張の論点として README や docs に記載すること。
